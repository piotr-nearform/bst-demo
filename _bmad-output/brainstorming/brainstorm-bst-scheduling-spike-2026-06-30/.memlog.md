- (technique) started Job to Be Done
- (insight by user) Spike's real job: establish ways of working KPMG + Nearform agree on, reusable to build the actual project after Discovery (we are pre-Discovery now)
- (insight by user) Hottest KPMG pushback = limited trust in AI building the app; mitigate with heavy test harness + automated multi-model reviews + human-in-the-loop (likely both)
- (idea by user) Secondary pushback: technology — KPMG may want Python over JS, and/or full Azure suite (Azure DevOps) instead of GitHub
- (idea by user) Accountability anchor: a human leaves the final PR approval and owns it at that point — net MORE accountability than today, not less; AI is tooling to streamline
- (idea by user) Put the PRD in version control: product DECISIONS become versioned and visible (when+where each was captured), not just the implementation
- (insight by user) Trade-off: the merge/approval decision gets harder (much more to review); multi-modal review is the tooling that supports the human reviewer
- (idea by user) Rubber-stamp risk isn't AI-specific; it exists in all SWE. Arguably trust a good multi-modal agentic review over some human reviewers
- (idea by user) Keep the pre-AI norm: a PR must be small enough for one human to read and understand it -> small vertical slices (caveat: may be a real blocker with AI tools that generate large changes)
- (idea by user) PR descriptions: simple+compact, reiterate the problem/AC from the ticket, and map exactly which tests cover each AC
- (insight by user) Good testing practice makes a PR readable: 'the test evidence is the product, it needs to read well'. Reviewer focuses on tests, skims implementation, watches sensitive files (package.json, infra, deps)
- (idea by user) Velocity: structured right, the product 'builds itself' and we're mostly observers. Slower than unsupervised AI (known, not the bar), but requirements-definition AND dev cycles are both much faster; with the right automation we power through stories
- (insight by coach) Convergence: small vertical slices is the keystone constraint — it serves BOTH accountability (human can genuinely review/own it) AND velocity (review stays cheap so it doesn't eat the AI speed gains)
- (technique) started One Feature Only
- (decision) Chosen single feature for the spike: the double-booking guard — assign a person to a job over a date range; system refuses any assignment that clashes with an existing one
- (idea by user) Non-trivial because of concurrency: two planners booking the same person at the same time mid-flight must be handled (race/conflict), not just a naive read-then-check. This is what lets the spike showcase HOW we work, not trivial CRUD
- (decision by user) Scope: ONLY the concurrent-clash edge for the spike — skip partial-overlap and touching-boundary cases. Target: wrap the whole spike in 1-2 days
- (technique) started Backcasting
- (decision by user) Step one tomorrow = write the first PRD line (PRD-first), and let the BMAD pipeline pull the React/Node/Azure scaffold into existence as the first story demands it — not scaffold-first
- (idea by user) Keep using the tools humans normally use: link the flow to real stories in Jira/Azure Boards even though code+PRD-in-version-control is the source of truth. Meet the client where they are
- (insight) SYNTHESIS: (1) Engineering hygiene = the sales pitch — a readable test-first PR is both the dev practice and the KPMG trust demo, same artifact. (2) The concurrent-clash edge is what forces the PO<->engineer spec negotiation, so the feature choice and the multi-role showcase are one decision. (3) Code-as-truth vs humans'-tools tension dissolves: PRD-in-VC canonical + Azure Boards as the human view + AB# link keeps them in sync free. (4) Small vertical slices is the keystone lever paying 3x: accountability + velocity + reviewability. CHOSEN DIRECTION: build the double-booking guard (concurrent-clash only, 1-2 days), PRD-first, demoed as one human-approved PR proving AI-authored-but-human-owned.
